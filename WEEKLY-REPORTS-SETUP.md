# Weekly Reports — Setup & Operations Guide

This is the **delta** you apply on top of the existing `SETUP.md` to turn on
**per-intern weekly summary reports**. It assumes the daily standup bot is
already working (Slack → Gemini → Sheets → GitHub Pages dashboard).

The feature lives on the `weekly-reports` branch. Once merged to `main` it runs
automatically; the manual work below is a few spreadsheet edits, one local test,
and (optionally) one cron-job.org trigger for on-time firing.

**Cost:** still $0/month — it reuses the same Gemini free tier, the same Google
Sheet, and the same GitHub Actions/Pages. **No new secrets are required.**

---

## 0 — What this feature does (and the one design decision behind it)

Every week the bot produces, **per intern**, a single technical paragraph
summarizing what that intern accomplished over the previous Monday–Friday. It is
generated by Gemini from the daily standup data and shown in a new **Weekly
reports** section on the dashboard.

### Why the "Yesterday" field was renamed to "Previous Workday"

The summary is built **only** from the field that used to be called *Yesterday*.
The problem: an intern logs what they did on a given day on the **next working
morning**. So Friday's work is never reported on Friday — it's reported the
following **Monday**. "Yesterday" is also wrong on Mondays (yesterday was
Sunday, a non-workday). The field is therefore renamed to **`Previous Workday`**,
and the standup question becomes *"What did you accomplish on your previous
workday?"*. A week then assembles like this:

| Work done on… | …is captured in the `Previous Workday` field of |
|---|---|
| Monday | Tuesday's entry |
| Tuesday | Wednesday's entry |
| Wednesday | Thursday's entry |
| Thursday | Friday's entry |
| **Friday** | **the *next* Monday's entry** |

### Why the job runs on **Tuesday**, not Monday

Because Friday's work only appears in the **next Monday's** entry, a complete
Mon–Fri week can't be assembled until that Monday's standup has been submitted
and polled. So the weekly job runs on the **new week's Tuesday morning** (by then
Monday's reply is recorded). Running it on Monday would always miss the previous
Friday. This is fixed in code — you don't configure it — but it's why the
schedule is Tuesday.

### Missing-data behavior

- If **some** workdays have no entry, the summary is built from whatever is
  present and **does not mention** the gaps.
- If **the entire week** is empty for an intern, no Gemini call is made; the row
  is stored with status `no_data` and the dashboard shows *"No work activity was
  logged for this intern during the reporting week."*

## 1 — Update the Google Sheet (one-time, ~5 minutes)

### 1.1 Rename the `Yesterday` header in every intern tab
For **each** per-intern tab (`Intern A`, `Intern B`, …), change the header in
**cell C1** from `Yesterday` to exactly:

```
Previous Workday
```

> Only the **header text** changes — the column keeps all its existing data, and
> the code reads the column by this header name, so the spelling must match
> exactly (capital P, capital W, single space). Existing daily rows are
> unaffected and remain visible in the dashboard's History table.

### 1.2 The `weekly` tab (auto-created — nothing to do, but here's what it is)
The job creates a tab named `weekly` automatically on its first run, with these
headers:

| A | B | C | D | E | F | G |
|---|---|---|---|---|---|---|
| `Week Start` | `Week End` | `Tab Name` | `Intern Name` | `Summary` | `Status` | `Generated At` |

You only need to create it by hand if your service account lacks permission to
add tabs — it shouldn't, since it was shared as **Editor** in `SETUP.md` Step 3.5.
If you do create it manually, add a tab named exactly `weekly` and type the seven
headers above into row 1.

> **`Tab Name`** in this tab matches the intern's per-intern tab name (the same
> value as the `Tab Name` column in the `config` tab), which is how each weekly
> summary is linked back to its intern on the dashboard.

|---|---|
| `GEMINI_API_KEY` | Generate the summary paragraph |

If you keep a local `.env`, no changes are needed there either — the same keys
already cover it.

## 3 — Test locally before relying on the schedule

From the project root, on the `weekly-reports` branch, with your `.env` filled in
(and `python-dotenv` installed as in `SETUP.md` Step 5.2):

```powershell
python scripts/post_weekly_report.py
```

**What to expect:**
- Console prints the reported week range and one line per intern ending in `ok`
  or `no_data`.
- A `weekly` tab appears in your spreadsheet (if it didn't exist), with one row
  per intern for the reported week.
- Re-running it **overwrites** those rows rather than duplicating (it's
  idempotent per week).

> **Tip for a meaningful first test:** the job summarizes the *previous* complete
> week. If your sheet only has a day or two of recent data, most interns will come
> back `no_data` — that's correct behavior, not a bug. To see a real paragraph,
> put a few sentences into the `Previous Workday` cells of the relevant dates
> (see the table in §0), or just wait until you have a real week of standups.

Then rebuild and preview the dashboard:

```powershell
python scripts/build_dashboard_data.py
python -m http.server -d docs 8000   # open http://localhost:8000
```

You should see a new **Weekly reports** section with a collapsible entry per
intern containing the paragraph(s).

## 4 — Turn it on in GitHub

The branch already includes `.github/workflows/weekly-report.yml`. Once it's on
your default branch (`main`), GitHub picks it up automatically.

### 4.1 Merge the branch
Merge `weekly-reports` into `main` (PR or fast-forward — your choice). No Pages
or repo settings need to change; the weekly workflow deploys to the same Pages
site as the poller and shares its `pages` concurrency group so the two never
clash.

### 4.2 Run it once by hand to confirm
1. **Actions** tab → **Weekly Report & Deploy** → **Run workflow** (it has a
   `workflow_dispatch` trigger).
2. Watch the run: it builds summaries, regenerates `docs/data.json`, and deploys
   Pages.
3. Reload the dashboard and confirm the **Weekly reports** section populated.

### 4.3 The automatic schedule
The workflow's built-in GitHub `schedule:` is:

```
cron: "0 5 * * 2"   # 10:00 PKT (UTC+5) every Tuesday
```

As explained in `notes.md`, GitHub's `schedule:` is best-effort and can be late
or dropped. For a once-a-week job a delay of a few hours is usually harmless, so
the GitHub schedule alone may be good enough. If you want it to fire dead-on
time, add the optional cron-job.org trigger in §5.

## 5 — (Optional) On-time firing with cron-job.org

This mirrors the hybrid trigger you already set up for the standup poster in
`SETUP.md` Step 9. **Reuse the same `GH_DISPATCH_TOKEN`** — it already covers
every workflow in the repo, so no new token is needed.

### 5.1 Build the endpoint URL
Same template as `SETUP.md` Step 9.2, with the workflow filename
`weekly-report.yml`. For the default repo it is:

```
https://api.github.com/repos/MAbdullah19/TimeSheet-Assistant/actions/workflows/weekly-report.yml/dispatches
```

Verify it (read-only, triggers nothing) before wiring it up:

```powershell
$token = "<GH_DISPATCH_TOKEN>"
Invoke-RestMethod "https://api.github.com/repos/MAbdullah19/TimeSheet-Assistant/actions/workflows/weekly-report.yml" -Headers @{ Authorization = "Bearer $token"; Accept = "application/vnd.github+json" }
```

Run the `Invoke-RestMethod` line as a **single line**. Expect
`name : Weekly Report & Deploy` and `state : active`. A `404` means a wrong
filename/owner/repo or an out-of-scope token; `401` a bad/expired token.

### 5.2 Create the cron-job.org job
1. Console → **Create cronjob**.
2. **Title:** `Trigger Weekly Report`.
3. **URL:** the endpoint from §5.1.
4. **Schedule:** custom —
   - **Minutes:** `0`.
   - **Hours:** `10` (→ 10:00 PKT; your account timezone is already
     `Asia/Karachi` from `SETUP.md` Step 9.3).
   - **Days of month / Months:** every (`*`).
   - **Days of week:** **Tuesday only** (untick everything except Tue).
5. **Advanced** (request settings) — identical to the other jobs:
   - **Request method:** `POST`
   - **Headers:**

     | Header | Value |
     |---|---|
     | `Accept` | `application/vnd.github+json` |
     | `Authorization` | `Bearer <GH_DISPATCH_TOKEN>` |
     | `X-GitHub-Api-Version` | `2022-11-28` |
     | `Content-Type` | `application/json` |

   - **Request body:** `{"ref":"main"}`
6. **Save**, then **Run now** to test — expect HTTP `204`, then a new **Weekly
   Report & Deploy** run in the Actions tab.

> **Double-fire is harmless.** If both the GitHub schedule and cron-job.org fire
> the same Tuesday, the job just regenerates the same week's rows in place
> (upsert by week + intern), and the shared `pages` concurrency group serializes
> the deploys.

## 7 — Changing the schedule or wording later

- **Run time / day:** edit the cron-job.org job (PKT) and/or the `cron:` line in
  `.github/workflows/weekly-report.yml` (UTC; PKT = UTC+5). If you move it off
  Tuesday, keep it **on or after Tuesday** of the new week — earlier and the
  previous Friday's data won't exist yet.
- **Summary tone/structure:** edit `SYSTEM_INSTRUCTION` in `core/weekly.py`.
- **Standup question wording:** edit the prompt text in
  `scripts/post_questions.py` (question 2). If you change what "Previous Workday"
  means, revisit the date mapping in `core/weekly.py`.

## 8 — Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Every intern returns `no_data` | No `Previous Workday` content for the reported week, or the header isn't exactly `Previous Workday` | Check §1.1 spelling; confirm the relevant dates have data (§0 table) |
| `WorksheetNotFound: weekly` on a manual sheet | You created `weekly` with a typo, or the service account can't add tabs | Name the tab exactly `weekly`; confirm Editor sharing (`SETUP.md` Step 3.5) |
| Summary is a plain run-on of the notes (not polished) | Gemini call failed and fell back | Re-check `GEMINI_API_KEY`; the fallback is intentional so a week is never lost |
| Weekly section empty on the dashboard | `data.json` predates the weekly run | Re-run **Weekly Report & Deploy** (it rebuilds `data.json`) |
| Friday's work never shows up | Job ran on Monday, or before Monday's poll | Ensure it runs **Tuesday** and the Monday poll has completed |

## 9 — What's intentionally *not* in this change

The larger **dashboard-as-a-multi-page-website** redesign (separate pages for the
daily board, per-intern history, and weekly reports instead of one long page) is
**out of scope** for this branch. The weekly data is already exposed in
`data.json` (each intern now carries a `weekly: [...]` array), so the future
multi-page UI can consume it without further backend work.
